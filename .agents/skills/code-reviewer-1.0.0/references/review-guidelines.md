## 检查规则概述

本审查工具通过静态代码分析方法，对代码进行全面的质量评估。检查内容涵盖：

1. **代码规范性**：文件命名、变量命名、代码格式等
2. **潜在Bug**：可能导致运行时错误的代码模式
3. **性能和安全**：性能问题和安全漏洞风险
4. **代码可读性**：代码长度、复杂度等可读性问题
5. **代码维护性**：TODO、FIXME等未完成项

## 严重性分级标准

### 严重 (Critical)
**定义**：可能导致功能错误、安全漏洞或系统崩溃的问题
**处理要求**：必须立即修复，阻止代码合并

**典型问题**：
- SQL注入漏洞
- 硬编码的密码或密钥
- 未初始化的变量使用
- 资源泄漏（文件、数据库连接等）
- 空指针解引用风险

### 一般 (Normal)
**定义**：影响代码质量、可维护性或可读性的问题
**处理要求**：建议在下次代码审查时修复

**典型问题**：
- 命名不规范
- 缺少必要的注释
- 存在调试代码（console.log等）
- 函数过长或复杂度过高
- 未完成的TODO/FIXME

### 优化 (Optimize)
**定义**：性能优化、代码风格或最佳实践建议
**处理要求**：可根据项目进度安排，不阻止代码合并

**典型问题**：
- 性能优化建议
- 代码风格改进
- 使用过时的API或语法
- 可以简化的代码结构

## 各语言特定检查项

### Python 检查项

**文件命名规范**
- 使用小写字母和下划线：`my_module.py`
- 避免使用连字符：`❌ my-module.py`

**变量命名规范**
- 普通变量：snake_case，如 `user_name`
- 常量：UPPER_SNAKE_CASE，如 `MAX_SIZE`
- 类名：PascalCase，如 `UserService`

**特定检查**
- 使用 `var` 关键字（Python不适用，但检查类似问题）
- 未导入的模块使用
- 异常处理不当（过于宽泛的except）
- 使用 `global` 或 `nonlocal` 时的警告

### JavaScript/TypeScript 检查项

**文件命名规范**
- 组件文件：kebab-case 或 camelCase
- `my-component.js` 或 `MyComponent.js`

**变量命名规范**
- 普通变量：camelCase，如 `userName`
- 常量：UPPER_SNAKE_CASE，如 `MAX_SIZE`
- 类名：PascalCase，如 `UserService`

**特定检查**
- 使用 `var` 声明变量（建议用 `let` 或 `const`）
- 存在 `console.log` 调试代码
- 未声明的变量使用
- 全局变量污染

### Java 检查项

**文件命名规范**
- 类文件：PascalCase，与类名一致
- `UserService.java`

**变量命名规范**
- 普通变量：camelCase，如 `userName`
- 常量：UPPER_SNAKE_CASE，如 `MAX_SIZE`
- 类名：PascalCase，如 `UserService`

**特定检查**
- 包名全小写
- 资源未关闭（IO流、数据库连接等）
- 异常处理不当

### C/C++ 检查项

**文件命名规范**
- 头文件：小写 + .h，如 `utils.h`
- 源文件：小写 + .c 或 .cpp，如 `utils.cpp`

**变量命名规范**
- 遵循项目约定，常见风格：
  - snake_case：`user_name`
  - camelCase：`userName`
  - 匈牙利命名法：`nCount`

**特定检查**
- 内存泄漏风险（malloc/new 与 free/delete 不匹配）
- 未初始化的指针使用
- 数组越界风险

## 代码规范性检查

### 文件命名规范

**检查内容**：
- 文件名是否符合语言约定
- 是否包含非法字符
- 大小写是否正确

**常见问题**：
- Python文件使用驼峰命名
- Java文件名与类名不一致
- 文件名包含空格或特殊字符

### 变量命名规范

**检查内容**：
- 变量命名是否符合语言约定
- 是否使用有意义的名称
- 避免缩写和单字母变量（除循环变量外）

**常见问题**：
- 变量名使用拼音
- 过度缩写：`usrNm` → `userName`
- 无意义命名：`temp`, `data`, `value`

### 注释完整性

**检查内容**：
- 注释覆盖率（注释行数 / 总代码行数）
- 公共函数和类是否有文档字符串
- 复杂逻辑是否有注释说明

**推荐标准**：
- 注释覆盖率 ≥ 20%（优秀）
- 注释覆盖率 ≥ 10%（良好）
- 注释覆盖率 < 10%（需改进）

## 潜在Bug检测

### 空指针风险

**检测模式**：
- Python: `if variable:` 而未检查 `variable is not None`
- JavaScript: 直接使用可能为 undefined 的变量
- Java: 直接使用可能为 null 的对象

**建议**：
- 显式检查 None/null
- 使用可选链操作符（JavaScript 2020+）
- 添加防御性编程

### 资源泄漏

**检测模式**：
- 文件打开后未关闭
- 数据库连接未释放
- 网络连接未关闭

**建议**：
- 使用上下文管理器（Python `with`）
- 使用 try-finally 确保资源释放
- 使用连接池管理数据库连接

### 未初始化变量

**检测模式**：
- 变量声明后未赋值就使用
- 函数参数默认值可能被错误使用

**建议**：
- 声明时初始化
- 检查变量的赋值路径
- 使用类型系统约束

## 性能优化建议

### 循环优化

**检测模式**：
- 嵌套循环（O(n²) 或更高复杂度）
- 循环内重复计算
- 循环内进行数据库查询

**建议**：
- 使用更高效的算法
- 将重复计算移到循环外
- 批量处理数据库操作

### SQL查询优化

**检测模式**：
- 字符串拼接构建SQL
- SELECT * 查询
- 缺少索引的查询（基于启发式）

**建议**：
- 使用参数化查询
- 只查询需要的字段
- 添加适当的索引
- 使用ORM框架

### 其他性能问题

**检测模式**：
- 大文件一次性读取到内存
- 同步阻塞操作
- 不必要的类型转换

**建议**：
- 流式处理大文件
- 使用异步操作
- 避免不必要的类型转换

## 代码可读性评估

### 代码行长度

**标准**：
- 推荐长度：≤ 80 字符
- 最大长度：≤ 120 字符
- 超过 120 字符：警告

**建议**：
- 将长行拆分为多行
- 使用合理的缩进
- 提取复杂表达式为变量

### 函数长度

**标准**：
- 推荐长度：≤ 30 行
- 可接受长度：≤ 50 行
- 超过 50 行：警告

**建议**：
- 拆分为更小的函数
- 每个函数只做一件事
- 提取重复逻辑

### 圈复杂度

**计算方式**：
- 每个分支（if, for, while, case）+1

**标准**：
- 低复杂度：1-10
- 中等复杂度：11-20
- 高复杂度：> 20

**建议**：
- 简化条件判断
- 使用早返回模式
- 提取复杂逻辑为独立函数

## 最佳实践建议

### 通用建议
1. **保持函数简短**：每个函数只做一件事
2. **有意义的命名**：使用描述性的名称
3. **及时注释**：解释"为什么"而不是"是什么"
4. **DRY原则**：不要重复自己
5. **SOLID原则**：遵循面向对象设计原则

### 语言特定建议
- **Python**: 遵循 PEP 8 规范，使用列表推导式和生成器
- **JavaScript**: 使用现代ES6+语法，避免全局变量
- **Java**: 遵循 Java 编码规范，合理使用异常
- **C/C++**: 注意内存管理，使用 RAII 模式

## 工具限制

本审查工具的局限性：
1. 仅进行静态分析，无法检测运行时问题
2. 无法理解复杂的业务逻辑
3. 可能存在误报和漏报
4. 对于大型项目，分析时间较长

**建议**：
- 结合单元测试和集成测试
- 使用多种静态分析工具
- 进行人工代码审查
- 定期重构和优化代码
